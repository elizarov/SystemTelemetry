System Telemetry
================

This project is a compact, real-time system monitoring dashboard designed for a small 800x480 secondary display,
presenting only the telemetry that can be collected cleanly through Windows APIs or vendor APIs.
It combines CPU load, temperature, clock, fan speed, and RAM usage with GPU load, temperature, clock, fan speed, and VRAM usage,
alongside network activity, multi-drive storage status, and a clock, all organized into a balanced panel layout.
The design emphasizes glanceable visuals with large load indicators, concise numeric stats, and minimal clutter.

Project Environment and Setup
=============================

The project runs on Windows 11 and is implemented in C++20 using Visual Studio 2022.
Use `devenv.cmd` to activate the development environment when needed.
Keep machine-specific compiler/tool paths in `devenv.cmd`, and document the expected `devenv.cmd` contract in `devenv.txt`.
Keep all sources in `src`.
Keep generated build outputs and temporary compiler files inside `build\` so the project root stays clean.
Keep transient compiler scratch files out of the top level of `build\`; route them into a dedicated `build\tmp\` area so the main build output folder stays readable.
Use a fresh per-build subdirectory under `build\tmp\` for `TMP` and `TEMP` so stale or externally locked toolchain scratch files do not break later builds.
Always build through `build.cmd`.

Requirements
============

Runtime requirements:

- Windows 11
- AMD Software: Adrenalin Edition, including the ADLX runtime, for AMD GPU telemetry

Development requirements:

- Visual Studio 2022 Build Tools

The application must not depend on LibreHardwareMonitor or OpenHardwareMonitor.

Configuration
=============

Make the system configurable for display placement and the subset of data sources that still require runtime choice.
Examples include:

- Display on which to show the system
- Relative position on the selected display
- Which network adapter to prefer
- Which drives to show
- Which Gigabyte motherboard fan channel to prefer when multiple tachometers are exposed

At runtime, the application must read `config.ini` from the same folder it was started from and write updates back to that same file.
The `src/config.ini` file in the source tree is only a checked-in example configuration template.
All runtime text such as config values, telemetry names, diagnostics, monitor identifiers, and UI strings must be stored internally as UTF-8 `std::string`.
Convert between UTF-8 `std::string` and UTF-16 `std::wstring` only at Windows API boundaries.
Configuration file I/O must use standard C++ stream primitives and read/write UTF-8 text directly without introducing a BOM during saves.
When `network.adapter_name` is left empty, auto-selection should prefer the active adapter that best represents the routed connection, favoring adapters with a usable default gateway and IPv4 address over host-only or otherwise unrouted virtual adapters.
When `vendor.gigabyte.fan_channel` is `0`, auto-selection should prefer the first Gigabyte tachometer channel that reports a valid RPM. Values `1` through `6` should force a specific Gigabyte fan channel when the board exposes it.

Command-line dump mode
======================

Support a `/dump` command-line switch. When present, the application must initialize telemetry using the same runtime `config.ini`, open `telemetry_dump.txt` beside the executable through a standard C++ output stream, write plain UTF-8 text without a BOM, pass that stream into telemetry collection for trace logging, write the final telemetry snapshot directly into that same stream, render that same final snapshot through the dashboard UI into an offscreen buffer, save it beside the executable as `telemetry_screenshot.png`, close outputs, and exit without starting the GUI or affecting any already-running dashboard instance.
The `/dump` trace must cover all diagnostics collection paths, not only vendor GPU integration.
Each trace line for telemetry or vendor API calls should include the relevant returned status/result codes and any key sampled values that help explain later failures or missing metrics.
Trace output should go through one shared trace wrapper that safely handles a null output stream and centralizes common trace-formatting helpers.
The human-readable dump snapshot should include the board-vendor provider status plus Gigabyte debug details such as detected board identity, helper path, controller type, selected CPU temperature sensor, requested and selected fan channel, sampled diagnostics, and any raw chip-register details when that provider exposes them.

Telemetry sources
=================

CPU telemetry must rely only on Windows-accessible metrics that are available in a straightforward way:

- CPU load
- CPU clock
- RAM usage
- Gigabyte motherboard CPU temperature and fan speed on the target Gigabyte X570 AORUS system through the installed Gigabyte SIV hardware-monitor stack, without LibreHardwareMonitor or OpenHardwareMonitor

CPU package power is not required and should not rely on LibreHardwareMonitor or OpenHardwareMonitor.

GPU telemetry must use Windows counters for generic metrics and vendor APIs for vendor-specific metrics:

- GPU load from Windows/PDH
- GPU dedicated VRAM usage from Windows/PDH
- GPU total dedicated VRAM capacity from Windows adapter information, or from the vendor API when that is the more reliable source for the active GPU
- AMD GPU temperature, clock, and fan speed from AMD's ADLX API

If AMD ADLX is unavailable or unsupported, the dashboard should continue running and leave AMD vendor metrics unavailable instead of blocking startup.
The application must not rely on WMI for telemetry, names, or dump diagnostics.

Vendor telemetry integrations should live in separate source files so each vendor implementation remains isolated.
Gigabyte motherboard fan telemetry should keep working when the Gigabyte board-specific provider is unavailable by leaving the CPU fan metric unavailable instead of blocking startup.
The Gigabyte motherboard telemetry path should identify Gigabyte boards through non-WMI Windows metadata, launch a local `GigabyteSivProbe.exe` helper beside the executable, have that helper discover the installed SIV location from the Windows registry, talk to the Gigabyte SIV hardware-monitor service over the `GigabyteEngine` IPC pipe, collect the available `HwRegister` fan RPM and temperature readings, and report provider diagnostics through `/dump`.
The Gigabyte helper must support a `/daemon` mode that keeps one SIV monitor session open and streams fresh fan and temperature snapshots continuously so the dashboard does not need to relaunch the helper process for each sample.

Size and placement
==================

Panel size is 800x480.
Place it as a top-level window at the monitor that is specified in configuration.
Make sure the monitor to show the panel is selected by its name, not number,
so it survives plugging/unplugging other monitors into the system.
Support interactive repositioning so the user can discover and copy the monitor
name plus the relative X/Y placement for configuration.

Window controls
===============

Add a popup menu on right-click with these actions:

- Move
- Bring On Top
- Update Config
- Exit

When Move is active, the dashboard window should follow the mouse cursor until
the user places it. While moving, show an overlay in the top-left corner with:

- Monitor friendly name
- Relative X/Y position on that monitor

The overlay content should be easy to copy into configuration.
The application must also read the saved relative X/Y coordinates from
configuration at startup and place the window accordingly.
Add an `Update Config` action that writes the current display identifier and
relative X/Y placement back to the config file while preserving all other
settings.

Tray behavior
=============

Create a system tray icon with the same popup menu as the dashboard window.
The tray menu must expose the same actions:

- Move
- Bring On Top
- Update Config
- Exit

The dashboard should use normal window Z-order behavior so other windows may
cover it. `Bring On Top` should raise the dashboard when it needs to be found.
Double-clicking the tray icon should perform the same `Bring On Top` action.

Single-instance behavior
========================

At most one dashboard instance may be running at a time.
Starting a new copy must close the already running copy, then continue startup
as the remaining instance.
The `/dump` command-line mode is exempt from this behavior and should run as an
independent one-shot command.

Style Summary
=============

Visual Theme
Background: pure black (#000000)
Primary text: white (#FFFFFF)
Accent color: cyan/blue (#00BFFF or similar)

Panels
Rounded rectangles with thin white borders
Consistent padding inside all panels
Subtle spacing between panels (8-12 px)

Typography
Primary numbers (load, time): large, bold
Labels: small, minimal (Temp, Clock, RAM, etc.)
Units: slightly dimmer than values

Visual Elements
Circular gauges for CPU/GPU load with the ring drawn in the track color and only the load-sized sliver overlaid in the shared usage color
Horizontal bars for usage
Mini graphs for network activity
Clean, minimal headers
CPU and GPU load gauges must use the same fill color for their used arc as storage and other occupancy/usage bars use for their filled portion.
Each top-level panel header must show a small monochrome icon derived from the dashboard sketch for CPU, GPU, Network, Storage, and Time, with those icon assets stored under `resources\` and compiled into the executable.
The application must define a custom app icon asset under `resources\`, embed it into the executable, and use that embedded icon for the executable and the tray icon instead of the default Windows icon.

Design Principles
High contrast for readability on small screen
Prioritize large, glanceable values
Avoid clutter
Consistent alignment across panels

===========================

Network

Download speed -> Down 24.1 MB/s
Graphs

Upload history
Download history
Footer
