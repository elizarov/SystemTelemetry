System Telemetry layout language specification

Overview
========

`config.ini` uses a compact INI-based layout language.

The language is static:

- layout is loaded from config
- coordinates are computed once after config load or reload
- rendering uses those precomputed coordinates until the next reload

The language is centered around the dashboard's current shape:

- the dashboard-level `cards` layout places cards through generic containers
- cards contain a small static composition of known widget kinds

Compact rules
=============

- weights are written as `name:weight(...)` or `name:weight`
- layout containers are written as `kind(...)`
- size pairs are written as `x,y`
- font specs are written as `face,size,weight`
- widget parameters are written inline as `widget(key=value)` or `widget(metric.ref)`
- omitted weights default to `1`
- whitespace around commas is ignored

Structure
=========

The language has two levels:

1. Dashboard-level geometry, card-placement layout, palette, and fonts in `[layout]`
2. Card-local title, icon, and content composition in `[card.<id>]`

Dashboard section
=================

`[layout]` owns the shared geometry, dashboard card-placement layout, palette, and fonts.

Supported geometry keys:

- `window = width,height`
- `outer_margin = pixels`
- `row_gap = pixels`
- `card_gap = pixels`
- `card_padding = pixels`
- `card_radius = pixels`
- `card_border = pixels`
- `header = height,icon_size,icon_gap`
- `content_gap = pixels`
- `column_gap = pixels`
- `metric_row_height = pixels`
- `metric_label_width = pixels`
- `metric_value_gap = pixels`
- `metric_bar_height = pixels`
- `widget_line_gap = pixels`
- `drive_row_height = pixels`
- `drive_label_width = pixels`
- `drive_percent_width = pixels`
- `drive_free_width = pixels`
- `drive_bar_gap = pixels`
- `drive_value_gap = pixels`
- `drive_bar_height = pixels`
- `throughput_axis_width = pixels`
- `throughput_header_gap = pixels`
- `throughput_read_label_width = pixels`
- `throughput_write_label_width = pixels`
- `throughput_graph_height = pixels`
- `gauge_preferred_size = pixels`

Supported dashboard layout key:

- `cards = rows(columns:weight(card:weight,...), ...)`

Example:

- `cards = rows(columns:3(cpu,gpu), columns:2(network:4,storage:9,time:3))`

Color keys stay in `[layout]` and use `#RRGGBB`:

- `background_color`
- `foreground_color`
- `accent_color`
- `panel_border_color`
- `muted_text_color`
- `track_color`
- `panel_fill_color`
- `graph_background_color`
- `graph_grid_color`
- `graph_axis_color`

Font keys stay in `[layout]`:

- `font.title = face,size,weight`
- `font.big = face,size,weight`
- `font.value = face,size,weight`
- `font.label = face,size,weight`
- `font.small = face,size,weight`

Card sections
=============

Each card section uses:

- `title = ...`
- `icon = ...`
- `layout = ...`

Recommended section names:

- `[card.cpu]`
- `[card.gpu]`
- `[card.network]`
- `[card.storage]`
- `[card.time]`

Future cards follow the same pattern:

- `[card.memory]`
- `[card.board]`
- `[card.weather]`

Icon names
==========

`icon` values are resource names, not file paths.

Supported icon names:

- `cpu`
- `gpu`
- `network`
- `storage`
- `time`

Layout expressions
==================

Supported layout kinds:

- `rows(...)`
- `columns(...)`
- `stack(...)`
- `stack_top(...)`
- `center(...)`

Meaning:

- `rows(...)` splits dashboard cards vertically into weighted row groups, with an optional row-group weight written as `rows:weight(...)`
- `columns(...)` splits content horizontally into weighted children, with child weights written as `child:weight`
- `stack(...)` splits content vertically into weighted children, with child weights written as `child:weight`
- `stack_top(...)` packs children from the top using their preferred heights and leaves any remaining space below
- `center(...)` creates a centered vertical stack

`drive_usage_list(...)` rows use `drive_row_height` as their packed row height when placed inside a top-packed stack.

Nested layout expressions are allowed.

Example:

- `columns(stack:5(throughput:4(storage.read),throughput:4(storage.write)), drive_usage_list:7(C,D,E))`

Widget names
============

Supported widget names:

- `text`
- `gauge`
- `metric_list`
- `throughput`
- `network_footer`
- `spacer`
- `drive_usage_list`
- `clock_time`
- `clock_date`

Widget parameters
=================

Widgets may bind data inline. For widgets that accept a list, the plain comma-separated body is passed through as that widget's parameter string.

Examples:

- `text(cpu.name)`
- `gauge(gpu.load)`
- `metric_list(cpu.temp,cpu.clock,cpu.fan,cpu.ram)`
- `throughput(network.upload)`
- `drive_usage_list(C,D,E)`

`spacer` reserves layout space without drawing content.

Supported metric references include:

- `cpu.name`
- `cpu.load`
- `cpu.temp`
- `cpu.clock`
- `cpu.fan`
- `cpu.ram`
- `gpu.name`
- `gpu.load`
- `gpu.temp`
- `gpu.clock`
- `gpu.fan`
- `gpu.vram`
- `network.upload`
- `network.download`
- `storage.read`
- `storage.write`

Drive and sensor selection
==========================

Rendering-oriented selection lives in the layout language when it changes what is drawn.

Example:

- `drive_usage_list(C,D,E)`

This defines a vertical list of drive-usage widgets and their order.

Consistency rules
=================

Shared absolute geometry belongs only in `[layout]`, including:

- margins
- row and card gaps
- border radius and border width
- card padding
- header metrics
- shared content spacing
- shared metric row height
- widget-specific widths and bar thicknesses when a hard-coded widget shape needs to stay configurable

Card sections define relative structure only:

- card title
- card icon
- layout expression
- widget parameters
- relative child weights using the `name:weight(...)` form

Static sizing rules
===================

Layout is resolved without using current telemetry values.

Static sizing rules:

- header height comes from config
- metric rows share one configured height
- metric bars, drive row columns, throughput plot chrome, and gauge preferred size come from config
- throughput sections follow the configured vertical rhythm
- centered time/date widgets use weighted slots from `center(...)`
- repeated lists such as drives divide their assigned area using the item count from config

Validation rules
================

- every dashboard container weight must be positive
- every card referenced from `cards` must have a matching `[card.<id>]` section
- every icon name must resolve to a supported embedded resource
- every widget name must resolve to a supported hard-coded widget kind
