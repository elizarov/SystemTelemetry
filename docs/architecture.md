## Runtime flows

- Startup flow: `src/app_main.cpp` opts into per-monitor-V2 DPI awareness, then parses `/trace`, `/dump`, `/screenshot`, `/save-config`, `/save-full-config`, `/reload`, `/fake`, `/layout`, `/default-config`, `/blank`, `/scale`, `/exit`, plus the internal elevated-helper switches for config-save, config-to-display, and auto-start updates through the diagnostics helper module. `/blank` selects the renderer's blank export mode and is rejected when combined with `/fake`. `/layout:<name>` overrides the active named layout in memory after config load and again after diagnostics reload. `/scale:<value>` overrides `display.scale` in memory after config load and again after diagnostics reload. `/default-config` suppresses the executable-side `config.ini` overlay and keeps the embedded config as the only runtime config source for that process. `/exit` routes into a headless diagnostics path that creates either the real telemetry runtime or the fake-file runtime, performs one update, optionally forces a config reload through the same runtime-reload helper used by the live UI, optionally writes the requested dump, screenshot, minimal-config, and full-config outputs at the requested paths, and exits without entering the single-instance GUI path. Without `/exit`, the normal dashboard UI path starts and any requested diagnostics outputs stay attached to that running instance.
- Telemetry flow: `DashboardApp` asks `TelemetryCollector` for a `SystemSnapshot`, then the window paints directly from that snapshot.
- Provider flow: `TelemetryCollector` owns both a vendor GPU provider and a motherboard-vendor provider. Windows collects generic metrics such as CPU load, CPU clock, GPU load, used VRAM, network, layout-configured drive capacity, system-wide storage read/write throughput, and time; total VRAM capacity comes from Windows adapter info as a fallback and may be overridden by a GPU vendor provider when the vendor API has a more reliable hardware-specific value, while named `board.temp.*` and `board.fan.*` metrics may be filled by a board-specific provider on the target Gigabyte system.
- No-WMI flow: the runtime telemetry path does not use WMI. The CPU name comes from native CPU brand-string discovery with a registry fallback, while GPU names fall back to built-in labels unless a vendor provider supplies a better name.
- Vendor coverage: AMD Radeon GPUs use ADLX through `src/gpu_amd_adl.cpp`, and this target Gigabyte motherboard uses the in-process SIV reflection path in `src/board_gigabyte_siv.cpp`. Additional vendors can be added as new provider source files without complicating the Windows-native collector.
- Diagnostics output flow: `src/app_diagnostics.cpp` owns a small diagnostics session helper.
  - Opens `telemetry_trace.txt` in append mode in the current working directory when `/trace` is enabled and passes that stream through the active telemetry runtime.
  - Separately overwrites `telemetry_dump.txt`, `telemetry_screenshot.png`, `telemetry_config.ini`, and `telemetry_full_config.ini` in the current working directory on each requested output refresh.
  - Emits dump files through `src/snapshot_dump.cpp` as machine-parseable snapshot text and renders screenshot exports through the effective runtime layout config and the same Direct2D and DirectWrite scene used by the live dashboard so headless output matches live styling and scale.
  - Emits minimal config exports through the same diff-against-target save path that `Save Config` uses, emits full config exports through the embedded-template export path, and refreshes requested outputs once per second in UI mode and once after the first telemetry update in `/exit` mode.
- Fake telemetry flow: `src/telemetry_runtime_fake.cpp` skips all live telemetry providers, reads the selected fake dump file through `src/snapshot_dump.cpp`, resolves fake network and storage selections from snapshot-derived candidate lists, exposes that parsed snapshot directly to the renderer, and reloads the file once per second so manual edits can drive repeatable UI testing. `/fake` defaults to `telemetry_fake.txt` in the current working directory, and `/fake:<path>` selects a different fake dump file. Required fake-file open or parse failures follow the shared diagnostics dialog policy so `/fake /exit` fails promptly under `/trace`.
- Diagnostics tracing flow: `src/telemetry.cpp` traces CPU, GPU, memory, network, storage capacity, storage throughput, per-drive activity, and PDH setup/refresh calls with returned Win32 or PDH statuses plus selected values; vendor providers such as `src/gpu_amd_adl.cpp` and `src/board_gigabyte_siv.cpp` add matching per-call result tracing for their API interactions.
- Network selection flow: `src/telemetry.cpp` combines `GetIfTable2` traffic counters with one `GetAdaptersAddresses` snapshot per refresh.
  - Filters candidates down to non-loopback, up adapters that expose an IPv4 address, then traces and ranks those adapters by routed-connection signals such as default-gateway presence, hardware/connector flags, and observed traffic before choosing the adapter to display.
  - Prefers exact case-insensitive alias or description matches when `network.adapter_name` is configured, then falls back to substring matching.
  - Falls back to the same auto-selection ranking when `network.adapter_name` is empty or no current IPv4 candidate matches it, and exposes the current candidate list with adapter-name-plus-IP labels for the popup menu.
- Trace ownership flow: `src/app_diagnostics.cpp` creates the diagnostics-session top-level trace wrapper for startup and output markers when `/trace` is enabled, `src/telemetry.cpp` owns a shared `Trace` instance for collector diagnostics, and vendor providers receive that same trace wrapper so all trace lines share one output policy and formatting helper set.
- Text model flow: config values, telemetry names, diagnostics, monitor identifiers, and UI text are stored as UTF-8 `std::string` in the app's own data structures. Wide strings are created only at Win32 or ADLX call sites through `src/utf8.cpp`.
- Config I/O flow: `src/config_parser.cpp` and `src/config_writer.cpp` read and write UTF-8 `config.ini` text through standard `std::ifstream` and `std::ofstream` primitives, while `src/config_resolution.cpp` owns active-layout selection and other derived config resolution helpers.
  - Strips an existing UTF-8 BOM on load.
  - Loads the embedded `resources/config.ini` blob through the Win32 resource API as the baseline config, then overlays executable-side values on top unless `/default-config` suppresses that overlay for the current process.
  - Parses the `[display]` active-layout and wallpaper settings, the `[network]` adapter selection, the `[board]` logical-board-metric-to-sensor mapping section, the `[metric_scales]` section, the dedicated per-widget sizing sections, the named `[layout.<name>]` geometry sections, the shared `[dashboard]` and `[card_style]` geometry sections, the shared `[colors]` and `[fonts]` sections, and the `[card.*]` layout expressions and widget bindings.
  - Replaces each parsed named-layout or card layout expression during overlay so executable-side config overrides embedded defaults without duplicating cards or widgets, then marks card-layout leaf nodes that reference another card id so the renderer can expand those refs later without flattening them during parse.
  - Leaves parsed runtime-selection fields in their persisted shape, then lets telemetry resolve the current network adapter and storage drive set at runtime so empty `[storage] drives` becomes the machine's currently available fixed drives only in the effective runtime config and saved output.
  - Keeps the embedded resource as the only shipped source of default layout, card, and widget definitions.
  - Builds minimal saves by loading the target INI through the normal overlay path, comparing that loaded config against the in-memory config field by field, and updating only the changed keys back into the original INI text.
  - Builds full exports by starting from the embedded template text and updating every maintained key with the current in-memory values.
- UI flow: `src/dashboard_app.cpp`, `src/dashboard_controller.cpp`, and `src/app_display_config.cpp` split the live UI into a thin shell plus application orchestration.
  - Tracks the current monitor DPI, resolves the effective render scale from either `display.scale` or the current monitor DPI, converts saved logical placement into physical pixels for window moves through that effective scale, seeds the first `CreateWindowExW` size and position from the configured monitor when available, listens for `WM_DPICHANGED`, `WM_DISPLAYCHANGE`, `WM_DEVICECHANGE`, and `WM_SETTINGCHANGE`, rebuilds renderer resources when the dashboard crosses onto a monitor with a different effective scale, applies the configured wallpaper through `IDesktopWallpaper`, and keeps a placement-watch timer armed while a configured monitor is missing.
  - Measures move-overlay fonts at paint time and lays the overlay out from scaled metrics while showing the current monitor scale, monitor name, and logical position.
  - Keeps `DashboardApp` focused on HWND lifecycle, paint invalidation, message dispatch, tray icon lifetime, move-mode presentation, renderer resource lifetime, the one-time startup load of the shared localization catalog, and the tracked Win32 layout-edit tooltip window that follows the same actionable edit target resolved by the layout-edit controller for cursor choice and mouse-down behavior, while `DashboardShellUi` owns the popup menu, layout-edit direct-entry dialogs, and their modal input suppression around the live shell.
  - Uses `DashboardController` as the application-layer owner for the active config, runtime instance, diagnostics session, menu option caches, save and reload actions, layout switching, config-to-display, network and storage selection, and layout-edit mode coordination.
  - Keeps persistence, diagnostics, auto-start, runtime update, and layout-edit mutations directly in `DashboardController`, while routing the config-to-display workflow through focused top-level helpers in `src/app_display_config.h`.
  - Uses `DashboardRenderer` for WIC-based panel-icon loading, shared render-type-aware layout resolution, static layout caching, the shared widget factory boundary, cached renderer assets, the live Direct2D and DirectWrite window frame, the move-mode overlay that is composited into that same frame, and the WIC-backed Direct2D screenshot-export renderer while the concrete widget rendering logic lives in `src/widget/*.cpp`.
  - Keeps gauge widgets on the same full-fidelity draw path as other repaints, but lets `src/widget/gauge.cpp` precompute the stable track path plus per-segment paths during layout-state resolution and lazily build only the one live cumulative fill path during draw so drag frames avoid repeated per-segment fill calls without introducing a separate drag renderer or raster-only shortcut.
  - Reuses the renderer's resolved separator-guide metadata to hit-test interactive layout-edit drags, while `src/layout_edit_service.cpp` owns guide-path lookup, reseeding editable drag weights from the current resolved child extents while excluding fixed-height row children and `vertical_spring` children from the reseeded total, named-layout/card-layout config mutation, and renderer-backed preview evaluation for iterative nested snap solving; those preview extent probes resolve only card and widget rectangles, while `src/layout_edit_controller.cpp` keeps a drag-session extent cache keyed by the candidate widget and attempted guide weights so repeated snap comparisons reuse the same renderer-backed preview extents within one drag.
  - Marks active container-guide drags in the renderer so relayout frames skip rebuilding widget-local guides, static anchors, and dynamic text anchors that are unused during that drag, but keeps the same full-fidelity widget draw path for drag repaints instead of maintaining a separate reduced drag-preview renderer; both the live shell and the layout-drag benchmark harness drive the same HWND-backed Direct2D and DirectWrite `DrawWindow` path for repaint timing, screenshot export reuses that same Direct2D and DirectWrite scene through a WIC-backed render target, and the full edit artifacts rebuild once when the drag ends.
  - Routes direct widget-value edits through `src/layout_edit_commands.cpp`, where the shared layout-edit parameter table applies field-local clamps for each supported metric-list, drive-usage, throughput, gauge, and font-size edit before `DashboardController` syncs the resulting config back into the renderer and runtime effective-config path, while `src/layout_edit_controller.cpp` owns the interactive multi-field drag clamps for widget guides whose valid range depends on other live config values, and the same parameter metadata plus the embedded shared localization catalog provide localized descriptions and font-spec tooltip formatting without duplicating guide-only and anchor-only parameter taxonomies.
  - Uses `DashboardMetricSource` to resolve metric names like `board.temp.cpu`, `board.fan.system`, `gpu.load`, and `storage.read` into the data shape each widget needs.
  - Uses `rows(...)` as the only vertical card-layout container, keeps direct `text`, `network_footer`, and `vertical_spacer(...)` children at fixed preferred heights in those rows, lets `vertical_spring` absorb remaining row height by weight for top-packed or centered compositions, sources widget geometry from the dedicated widget config sections, expands card-layout references lazily during layout resolution with cycle protection, and suppresses header reservation entirely for cards whose title and icon are both omitted.
  - Routes popup-menu building, popup command dispatch, custom-scale prompting, and the direct layout-edit value, weight, and font dialogs through `src/dashboard_shell_ui.cpp` so the shell message loop does not carry the modal dialog procedures and menu command switch directly.
  - Builds the `Layout` submenu from the configured named layouts, appends each layout's optional description after its aspect-ratio name, shows the active layout with a radio check, and switches the live renderer, effective config, and window size immediately when the user selects a different layout.
  - Builds the `Scale` submenu from the default entry, the maintained preset percentages, and the current custom percentage when present, shows the active effective selection with a radio check, opens a small modal dialog for `Custom...`, and reapplies the live renderer scale and window size immediately when the user changes it.
  - Builds the `Network` submenu from the telemetry runtime's current adapter candidates, labels each entry with the same adapter-name-plus-IP footer text used by the dashboard, shows the runtime-selected adapter with a radio check, and applies the chosen adapter name immediately when selected.
  - Builds the `Storage drives` submenu from the telemetry runtime's shared storage-drive candidate list, labels each entry as drive letter plus optional volume label plus total size, shows the currently selected `[storage] drives` with checkbox state, and reapplies the renderer plus telemetry immediately when a drive is toggled.
  - Routes `WM_DEVICECHANGE`, `WM_SETTINGCHANGE`, and `WM_DISPLAYCHANGE` through an explicit telemetry-selection refresh-and-sample path so adapter and drive resolution rerun only on initialization, config changes, or topology-change triggers instead of on the shared 500 ms metrics timer, and the next rendered snapshot already reflects the newly resolved targets.
  - Builds the `Config To Display` submenu from the currently enumerable monitors, shows each monitor as friendly name plus physical resolution, enables monitors whose physical resolution matches the active layout's aspect ratio, and when selected computes and saves the fitted explicit `display.scale`, writes `telemetry_blank.png` with that same fitted scale, updates `config.ini`, and applies wallpaper in one local or elevated flow.
  - Tears down the current telemetry runtime before `Reload Config` creates a replacement, reapplies executable-side `config.ini`, reapplies any configured monitor wallpaper, recomputes static card and widget rectangles, and repaints immediately.
  - Moves without pre-sizing when reload restores placement onto a monitor with a different DPI scale so the `WM_DPICHANGED` suggestion can resize the window without double-scaling the restored bounds.
- Auto-start flow: `src/app_autostart.cpp` reads the machine-wide `HKLM\Software\Microsoft\Windows\CurrentVersion\Run\SystemTelemetry` value to decide whether the popup-menu toggle is checked, compares that stored command path against the current executable path case-insensitively after trimming quotes, writes or removes the value when toggled, and relaunches the same executable with `runas` plus an internal `/set-autostart on|off` helper switch when direct registry writes are denied.
- Embedded config flow: `resources/config.ini` is the checked-in self-documented template and is compiled into the executable; the running app reads that embedded copy first, overlays any executable-side `config.ini` unless `/default-config` is present, uses that same overlay path as the comparison baseline for minimal saves, and uses the embedded template text directly for full-config exports.
- Elevated config-save flow: `src/app_diagnostics.cpp` probes whether the current process can write the executable-side `config.ini`; if not, `Save Config` writes the effective settings to a temporary INI under `%TEMP%`, relaunches the same executable with `/save-config <temp-path>` using the `runas` verb, waits for that elevated helper instance to rewrite the live `config.ini`, and then removes the temporary handoff file.
- Virtualization behavior: because the executable carries an explicit application manifest, Windows should not redirect protected-location config reads or writes into `AppData\Local\VirtualStore`; non-elevated writes to `Program Files` should fail normally and then follow the explicit elevation path.

## Build and deployment

- Build flow: `build.cmd` boots the VS environment via `devenv.cmd`, redirects `TMP` and `TEMP` into a fresh per-build subdirectory under the user's temp area (falling back to `build\tmp\` only when no user temp path is available), re-creates `build\cmake\` when an older cache uses a different generator, configures `build\cmake\` through `cmake -S -B -G "Ninja Multi-Config" -DCMAKE_EXPORT_COMPILE_COMMANDS=ON`, then builds that tree through `cmake --build`.
  - Keeps generated Ninja files, dependency state, object files, and `compile_commands.json` under `build\cmake\` while placing the final executable under `build\` and leaving debug-symbol outputs there for debug-style configs.
  - Lets Ninja and the MSVC toolchain perform incremental per-source recompilation.
  - Compiles the Win32 app target from the split dashboard shell, controller, service, telemetry, renderer, diagnostics, the minimal vendored ADLX subset, and `resources/SystemTelemetry.rc`.
  - Builds `src/board_gigabyte_siv.cpp` separately as a Common Language Runtime-enabled object library so it can bridge into the Gigabyte .NET assemblies.
  - Resolves `mscoree.lib` from the Visual Studio developer environment through `NETFXSDKDir` and disables the linker manifest because the `.rc` file already embeds the maintained application manifest.
