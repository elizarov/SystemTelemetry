# System Telemetry architecture

## Source map

- `docs/`: project documentation. `docs/specifications.md` tracks user-visible dashboard behavior, `docs/diagnostics.md` tracks diagnostics CLI behavior and output requirements, `docs/layout.md` tracks configuration-language syntax and examples, `docs/project.md` tracks project environment and engineering constraints, and `docs/architecture.md` tracks structural and code-organization details.
- `src/main.cpp`: Win32 app entry point, single-instance startup, command-line diagnostics handling for `/trace`, `/dump`, `/screenshot`, `/save-config`, `/save-full-config`, `/reload`, `/fake`, `/layout`, `/edit-layout`, `/default-config`, `/blank`, `/scale`, `/exit`, and the internal elevated helper switches, locale-stable fractional `/scale` parsing, monitor identification, per-monitor DPI handling, tray/menu actions including `Config To Display`, a shared 500 ms timer-driven refresh loop, executable-relative `config.ini` path resolution, elevation-aware `Save Config` and auto-start toggles, working-directory-relative diagnostics Save-dialog exports and CLI default paths, configured per-monitor wallpaper application through `IDesktopWallpaper`, and orchestration of the telemetry runtime plus the dedicated dashboard renderer module.
- `src/config.h` + `src/config_schema.h` + `src/config.cpp`: runtime config model, schema metadata helpers, and load/save logic for executable-relative `config.ini`.
  - Loads embedded resource defaults first, then overlays executable-side config values.
  - Parses the `[display]`, `[network]`, `[storage]`, and `[board]` runtime sections, the dedicated `[metric_scales]` section, the per-widget sections named exactly after the widget (`[metric_list]`, `[drive_usage_list]`, `[throughput]`, `[gauge]`, `[text]`, `[network_footer]`, `[clock_time]`, and `[clock_date]`), the named `[layout.<name>]` sections, the shared `[dashboard]`, `[card_style]`, `[colors]`, and `[fonts]` sections, and the `[card.*]` layout language including the generic dashboard `cards = rows(...)` tree with `name:weight(...)` weight syntax.
  - Represents each multi-value config line as one composite config field, including `display.position`, `layout.window`, and the grouped `[fonts]` entries.
  - Declares schema metadata beside config fields for fixed sections and dynamic `[layout.<name>]` plus `[card.<id>]` sections, and uses one metadata-owned binding list per config owner with recursive bindings for nested config owners such as `AppConfig.layout` so generic section walkers can handle fixed, dynamic, and nested-recursive sections from the same list while section names, key names, inferred codecs, member pointers, collection keys, and the maintained `layout.` and `card.` prefixes stay in one place.
  - Uses a light macro layer in `src/config_schema.h` so persisted fields in `src/config.h` declare their member storage and schema metadata on the same line, section descriptors are generated from those reflected field declarations, and codec selection comes from type-to-codec and section-owner-to-section-codec traits with local specializations for project-specific config value types and custom section handlers such as `[board]`.
  - Keeps section names, key names, card ids, container names, widget names, icon names, and metric references case-sensitive so differently cased identifiers remain distinct and undocumented aliases or legacy spellings are rejected.
  - Carries segmented gauge geometry in `[gauge]`, derives the gauge's bottom-centered opening from `sweep_degrees`, derives metric-list and drive-usage row heights from measured fonts plus config keys, derives the rendered drive list from `[storage] drives`, derives requested named board sensors from layout bindings, resolves one active named layout into the renderer-facing layout structure, and supports both minimal config saves against the live INI text and full exports against the embedded template while preserving surrounding INI comments and structure.
  - Keeps config structs at neutral zero/empty defaults so shipped fonts, colors, layout sizes, widget sizes, and metric-list scale ceilings come only from the embedded `resources/config.ini` template plus any executable-side `config.ini` overlay.
  - Exposes the `[colors]` palette entries `graph_marker_color` for throughput guide and time-marker lines and `layout_guide_color` for interactive layout-edit separators.
- `src/dashboard_metrics.h` + `src/dashboard_metrics.cpp`: dashboard metric-binding layer that adapts `SystemSnapshot` into widget-oriented data shapes.
  - Resolves text values, gauge data, metric-list rows, throughput series, network footer text, clock text, and storage drive rows by metric name.
  - Uses one shared snapshot-level retained-history-series collection for metric-list rows, load gauges, and throughput widgets so every recent-history consumer shares the same path.
  - Applies retained normalized history ratios for peak ghosts, uses `AppConfig.metricScales` for every non-capacity metric-list normalization ceiling, smooths retained raw throughput history with a two-reading simple moving average before graph scaling and rendering, uses that same latest averaged point for each throughput value readout, and attaches one snapshot-time phase so 10-second vertical markers line up across all throughput plots.
  - Uses 5 MB/s horizontal guide spacing through an averaged graph max of 50 MB/s and 50 MB/s above that range for storage plots.
  - Converts per-drive MB/s samples into per-column read and write activity shares for the drive-usage indicator stack.
- `src/dashboard_renderer.h` + `src/dashboard_renderer.cpp`: dashboard layout and widget renderer module.
  - Owns panel icon loading, UI font creation, font measurement, static layout resolution, widget binding parsing, and GDI/GDI+ drawing for cards and supported widget kinds.
  - Computes metric-list and drive-usage header/row heights from measured font metrics plus widget-section keys, measures fixed-width text columns such as throughput labels, throughput axis labels, drive-letter labels, and drive-percent labels from configured fonts plus widget-section padding, and applies dedicated drive-list column-gap keys where the row needs asymmetric spacing.
  - Keeps renderer geometry in logical units, then applies a render-scale factor derived from monitor DPI or headless `/scale` before layout resolution so fonts, spacing, window size, and screenshot dimensions scale together into native device pixels.
  - Supports a renderer-owned blank mode that keeps static dashboard chrome and selected static labels while suppressing dynamic widget paint for background screenshot exports.
  - Keeps container names, widget names, card ids, and icon names case-sensitive so only documented spellings resolve to widget kinds, card placements, and embedded icon resources.
  - Resolves card and widget rectangles first, then scans the resolved gauge widgets to compute one shared fitted gauge radius for the active layout so every gauge render uses the same size while still respecting the tightest gauge slot.
  - Renders metric and drive bars as antialiased pill capsules, draws the drive-usage widget as a six-column table with a dedicated header band and stacked read/write activity segments, overlays translucent recent-peak capsule markers on metric rows, renders gauges as segmented annular slices with retained peak ghosts and whole-segment fill behavior, and reserves the configured plot stroke width as top inset for throughput plots so graph-max values stay inside the chart area.
  - Treats each widget binding as one generic primary parameter string whose meaning belongs to the hard-coded widget type, such as metric names for `text(...)`, `gauge(...)`, and `throughput(...)` or metric refs with optional `=Label` overrides for `metric_list(...)`, while `drive_usage_list` reads its rows from `[storage] drives`.
  - Pulls gauge, throughput, footer, clock, metric-list, drive-usage, and text chrome sizing from dedicated config sections, emits layout-computation traces when a diagnostics trace stream is attached, and records resolved draggable separator guides for `rows(...)` and `columns(...)` layout editing.
- `src/utf8.h` + `src/utf8.cpp`: shared UTF-8 and UTF-16 conversion helpers used only where the UTF-8-first runtime model crosses Win32 or vendor API boundaries.
- `src/trace.h` + `src/trace.cpp`: shared trace wrapper around an optional output stream plus common trace-formatting helpers for Win32, PDH, ADLX, boolean, and numeric diagnostic fields.
- `src/telemetry.h` + `src/telemetry.cpp`: shared telemetry snapshot types plus the `TelemetryCollector` implementation.
  - Gathers Windows-native CPU brand-string, PDH, IP Helper, disk-capacity, system-wide disk-throughput, per-drive logical-disk throughput, and clock data behind a small public API.
  - Refreshes every live metric source on the shared 500 ms UI timer after initialization so CPU, GPU, memory, network, storage throughput, per-drive storage activity, drive capacity, and clock stay aligned to one snapshot cadence.
  - Retains one 60-sample retained-history-series collection for load gauges, throughput plots, metric-list rows such as RAM, VRAM, temperature, clock, fan, and requested board sensors so all recent-history consumers share the same 30-second plumbing.
  - Uses the same config-driven metric-scale ceilings for normalized ratio series, while throughput series keep raw MB/s values for graph smoothing and scaling.
  - Keeps the exact-match series-ref-to-index map on `SystemSnapshot` and lazily registers retained-history series on first write so repeated updates avoid rescanning the retained-history vector.
  - Stores CPU board temperatures and fans in the named `board.temperatures` and `board.fans` collections.
- `src/telemetry_runtime.h` + `src/telemetry_runtime.cpp`: shared runtime abstraction plus the real telemetry runtime wrapper that adapts `TelemetryCollector` to the UI/diagnostics host flow. The same runtime layer also centralizes whether fake-runtime initialization failures are allowed to show modal dialogs so `/trace` diagnostics paths stay non-blocking, and keeps a UI-overridable effective config view for exports such as screenshots after live layout or placement changes.
- `src/telemetry_runtime_fake.cpp`: fake telemetry runtime implementation that reloads the selected fake dump file, defaulting to `telemetry_fake.txt` in the current working directory, through the dump parser once per second for repeatable UI testing.
- `src/snapshot_dump.h` + `src/snapshot_dump.cpp`: versioned text dump serializer/parser for the fake-runtime snapshot payload, including named board metrics, the shared retained-history-series collection for throughput and recent-peak widgets, drive rows with per-drive read/write MB/s, and full dumped local time; the same format powers both `/dump` exports and `/fake` imports.
- `src/gpu_vendor.h` + `src/gpu_vendor.cpp`: vendor GPU provider abstraction plus factory selection.
- `src/gpu_amd_adl.cpp`: AMD GPU provider implementation using AMD ADLX for GPU name, temperature, clock, and fan speed.
- `src/board_vendor.h` + `src/board_vendor.cpp`: motherboard-vendor provider abstraction plus factory selection. The shared sample carries board identity, provider diagnostics, requested sensor names, and named metrics that the dashboard and dump path use.
- `src/board_gigabyte_siv.cpp`: Gigabyte motherboard provider implementation.
  - Detects Gigabyte baseboards from registry-backed firmware metadata.
  - Discovers the installed SIV directory from the uninstall registry and resolves vendor assembly dependencies from that directory.
  - Loads the required Gigabyte assemblies in-process through a mixed-mode C++/CLI bridge.
  - Initializes `Gigabyte.Engine.EnvironmentControl.HardwareMonitor.HardwareMonitorControlModule` against the `HwRegister` source through reflection.
  - Maps layout-requested logical board fan and temperature metrics to SIV sensor titles through the parsed `[board]` config section without any helper process or text-protocol layer.
- `resources/`: embedded UI assets, including the panel header icons extracted from the design sketch, the custom app icon, the self-documented embedded `config.ini` template, and the `.rc` plus resource-id definitions used to compile them into the executable.
- `resources/SystemTelemetry.manifest`: embedded Win32 application manifest that keeps the app at `asInvoker`, disables legacy file virtualization so config I/O targets the real executable-side `config.ini`, and opts the process into per-monitor DPI-aware rendering.
- `install.cmd`: elevated deployment script that builds via `build.cmd`, installs `build\SystemTelemetry.exe` into `C:\Program Files\SystemTelemetry`, and leaves automatic startup to the runtime popup-menu toggle.
- `CMakeLists.txt`: single maintained build graph for the Win32 app plus the mixed-mode Gigabyte board-provider object library.
  - Defines the native source list once and keeps target outputs rooted in `build\`.
  - Lets MSBuild handle per-translation-unit incremental recompilation and header dependency tracking.
  - Compiles `src/board_gigabyte_siv.cpp` with Common Language Runtime support so it can call the vendor .NET assemblies in-process.
  - Links the required CLR host support library from the active Windows SDK.
- `src/vendor/adlx/...`: vendored AMD ADLX headers/helper/runtime glue used by the AMD GPU provider.
- `devenv.cmd` + `devenv.md`: local build-environment bootstrap plus its machine-specific setup contract; `devenv.cmd` prepares the Visual Studio x64 toolchain that `build.cmd` and the mixed-mode Gigabyte board-provider build use, while `devenv.md` documents that requirement for future machines.

## Runtime flows

- Startup flow: `wWinMain` opts into per-monitor-V2 DPI awareness, then parses `/trace`, `/dump`, `/screenshot`, `/save-config`, `/save-full-config`, `/reload`, `/fake`, `/layout`, `/default-config`, `/blank`, `/scale`, `/exit`, plus the internal elevated-helper switches for config-save and auto-start updates. `/blank` selects the renderer's blank export mode and is rejected when combined with `/fake`. `/layout:<name>` overrides the active named layout in memory after config load and again after diagnostics reload. `/default-config` suppresses the executable-side `config.ini` overlay and keeps the embedded config as the only runtime config source for that process. `/exit` routes into a headless diagnostics path that creates either the real telemetry runtime or the fake-file runtime, performs one update, optionally forces a config reload through the same runtime-reload helper used by the live UI, optionally writes the requested dump, screenshot, minimal-config, and full-config outputs at the requested paths, and exits without entering the single-instance GUI path. Without `/exit`, the normal dashboard UI path starts and any requested diagnostics outputs stay attached to that running instance.
- Telemetry flow: `DashboardApp` asks `TelemetryCollector` for a `SystemSnapshot`, then the window paints directly from that snapshot.
- Provider flow: `TelemetryCollector` owns both a vendor GPU provider and a motherboard-vendor provider. Windows collects generic metrics such as CPU load, CPU clock, GPU load, used VRAM, network, layout-configured drive capacity, system-wide storage read/write throughput, and time; total VRAM capacity comes from Windows adapter info as a fallback and may be overridden by a GPU vendor provider when the vendor API has a more reliable hardware-specific value, while named `board.temp.*` and `board.fan.*` metrics may be filled by a board-specific provider on the target Gigabyte system.
- No-WMI flow: the runtime telemetry path does not use WMI. The CPU name comes from native CPU brand-string discovery with a registry fallback, while GPU names fall back to built-in labels unless a vendor provider supplies a better name.
- Vendor coverage: AMD Radeon GPUs use ADLX through `src/gpu_amd_adl.cpp`, and this target Gigabyte motherboard uses the in-process SIV reflection path in `src/board_gigabyte_siv.cpp`. Additional vendors can be added as new provider source files without complicating the Windows-native collector.
- Diagnostics output flow: `src/main.cpp` owns a small diagnostics session helper.
  - Opens `telemetry_trace.txt` in append mode in the current working directory when `/trace` is enabled and passes that stream through the active telemetry runtime.
  - Separately overwrites `telemetry_dump.txt`, `telemetry_screenshot.png`, `telemetry_config.ini`, and `telemetry_full_config.ini` in the current working directory on each requested output refresh.
  - Emits dump files through `src/snapshot_dump.cpp` as machine-parseable snapshot text and renders screenshot exports through the effective runtime layout config so headless output matches dashboard styling.
  - Emits minimal config exports through the same diff-against-target save path that `Save Config` uses, emits full config exports through the embedded-template export path, and refreshes requested outputs once per second in UI mode and once after the first telemetry update in `/exit` mode.
- Fake telemetry flow: `src/telemetry_runtime_fake.cpp` skips all live telemetry providers, reads the selected fake dump file through `src/snapshot_dump.cpp`, exposes that parsed snapshot directly to the renderer, and reloads the file once per second so manual edits can drive repeatable UI testing. `/fake` defaults to `telemetry_fake.txt` in the current working directory, and `/fake:<path>` selects a different fake dump file. Required fake-file open or parse failures follow the shared diagnostics dialog policy so `/fake /exit` fails promptly under `/trace`.
- Diagnostics tracing flow: `src/telemetry.cpp` traces CPU, GPU, memory, network, storage capacity, storage throughput, per-drive activity, and PDH setup/refresh calls with returned Win32 or PDH statuses plus selected values; vendor providers such as `src/gpu_amd_adl.cpp` and `src/board_gigabyte_siv.cpp` add matching per-call result tracing for their API interactions.
- Network selection flow: `src/telemetry.cpp` combines `GetIfTable2` traffic counters with one `GetAdaptersAddresses` snapshot per refresh.
  - Filters candidates down to non-loopback, up adapters that currently expose an IPv4 address, then traces and ranks those adapters by routed-connection signals such as default-gateway presence, hardware/connector flags, and observed traffic before choosing the adapter to display.
  - Prefers exact case-insensitive alias or description matches when `network.adapter_name` is configured, then falls back to substring matching.
  - Falls back to the same auto-selection ranking when `network.adapter_name` is empty or no current IPv4 candidate matches it, and exposes the current candidate list with adapter-name-plus-IP labels for the popup menu.
- Trace ownership flow: `src/main.cpp` creates the diagnostics-session top-level trace wrapper for startup and output markers when `/trace` is enabled, `src/telemetry.cpp` owns a shared `Trace` instance for collector diagnostics, and vendor providers receive that same trace wrapper so all trace lines share one output policy and formatting helper set.
- Text model flow: config values, telemetry names, diagnostics, monitor identifiers, and UI text are stored as UTF-8 `std::string` in the app's own data structures. Wide strings are created only at Win32 or ADLX call sites through `src/utf8.cpp`.
- Config I/O flow: `src/config.cpp` reads and writes UTF-8 `config.ini` text through standard `std::ifstream` and `std::ofstream` primitives.
  - Strips an existing UTF-8 BOM on load for compatibility.
  - Loads the embedded `resources/config.ini` blob through the Win32 resource API as the baseline config, then overlays executable-side values on top unless `/default-config` suppresses that overlay for the current process.
  - Parses the `[display]` active-layout and wallpaper settings, the `[network]` adapter selection, the `[board]` logical-board-metric-to-sensor mapping section, the `[metric_scales]` section, the dedicated per-widget sizing sections, the named `[layout.<name>]` geometry sections, the shared `[dashboard]` and `[card_style]` geometry sections, the shared `[colors]` and `[fonts]` sections, and the `[card.*]` layout expressions and widget bindings.
  - Replaces each parsed named-layout or card layout expression during overlay so executable-side config overrides embedded defaults without duplicating cards or widgets, then marks card-layout leaf nodes that reference another card id so the renderer can expand those refs later without flattening them during parse.
  - Keeps the embedded resource as the only shipped source of default layout, card, and widget definitions.
  - Builds minimal saves by loading the target INI through the normal overlay path, comparing that loaded config against the current in-memory config field by field, and updating only the changed keys back into the original INI text.
  - Builds full exports by starting from the embedded template text and updating every maintained key with the current in-memory values.
- UI flow: `DashboardApp` owns the native window, tray icon, popup menu, move mode overlay, layout-edit overlay and drag handling, config reload/save actions, one-off diagnostics full-config, dump, and screenshot Save-dialog actions, custom app-icon loading for the window and tray, tray double-click bring-to-front handling, and the active telemetry runtime.
  - Tracks the current monitor DPI, converts saved logical placement into physical pixels for window moves, seeds the first `CreateWindowExW` size and position from the configured monitor when available, listens for `WM_DPICHANGED`, `WM_DISPLAYCHANGE`, `WM_DEVICECHANGE`, and `WM_SETTINGCHANGE`, rebuilds renderer resources when the dashboard crosses onto a monitor with a different scale, applies the configured wallpaper through `IDesktopWallpaper`, and keeps a placement-watch timer armed while a configured monitor is missing.
  - Measures move-overlay fonts at paint time and lays the overlay out from scaled metrics while showing the current monitor scale, monitor name, and logical position.
  - Uses `DashboardRenderer` for panel-icon loading, UI font creation, static layout caching, and GDI/GDI+ drawing for cards and generic widget kinds such as text, gauge, metric-list, throughput, drive-usage, clock, footer, and spacer widgets.
  - Reuses the renderer's resolved separator-guide metadata to hit-test interactive layout-edit drags, updates the underlying active named layout or card layout weights in memory, and keeps `Save Config` aligned with those live edits through the runtime effective-config path.
  - Uses `DashboardMetricSource` to resolve metric names like `board.temp.cpu`, `board.fan.system`, `gpu.load`, and `storage.read` into the data shape each widget needs.
  - Supports both stretch-to-fill vertical stacks and top-packed `stack_top(...)` containers, sources widget geometry from the dedicated widget config sections, expands card-layout references lazily during layout resolution with cycle protection, and suppresses header reservation entirely for cards whose title and icon are both omitted.
  - Builds the `Layout` submenu from the configured named layouts, shows the active layout with a radio check, and switches the live renderer, effective config, and window size immediately when the user selects a different layout.
  - Builds the `Network` submenu from the telemetry runtime's current adapter candidates, labels each entry with the same adapter-name-plus-IP footer text used by the dashboard, shows the runtime-selected adapter with a radio check, and applies the chosen adapter name immediately when selected.
  - Builds the `Storage drives` submenu from the telemetry runtime's current active fixed and removable drives, labels each entry as drive letter plus optional volume label plus total size, shows the currently selected `[storage] drives` with checkbox state, and reapplies the renderer plus telemetry immediately when a drive is toggled.
  - Builds the `Config To Display` submenu from the currently enumerable monitors, shows each monitor as friendly name plus physical resolution, greys out monitors whose physical resolution does not match the dashboard's DPI-scaled window size, and when selected writes `telemetry_blank.png`, updates `config.ini`, and applies wallpaper in one local or elevated flow.
  - Tears down the current telemetry runtime before `Reload Config` creates a replacement, reapplies executable-side `config.ini`, reapplies any configured monitor wallpaper, recomputes static card and widget rectangles, and repaints immediately.
  - Moves without pre-sizing when reload restores placement onto a monitor with a different DPI scale so the `WM_DPICHANGED` suggestion can resize the window without double-scaling the restored bounds.
- Auto-start flow: `src/main.cpp` reads the machine-wide `HKLM\Software\Microsoft\Windows\CurrentVersion\Run\SystemTelemetry` value to decide whether the popup-menu toggle is checked, compares that stored command path against the current executable path case-insensitively after trimming quotes, writes or removes the value when toggled, and relaunches the same executable with `runas` plus an internal `/set-autostart on|off` helper switch when direct registry writes are denied.
- Embedded config flow: `resources/config.ini` is the checked-in self-documented template and is compiled into the executable; the running app reads that embedded copy first, overlays any executable-side `config.ini` unless `/default-config` is present, uses that same overlay path as the comparison baseline for minimal saves, and uses the embedded template text directly for full-config exports.
- Elevated config-save flow: `src/main.cpp` probes whether the current process can write the executable-side `config.ini`; if not, `Save Config` writes the effective settings to a temporary INI under `%TEMP%`, relaunches the same executable with `/save-config <temp-path>` using the `runas` verb, waits for that elevated helper instance to rewrite the live `config.ini`, and then removes the temporary handoff file.
- Virtualization behavior: because the executable carries an explicit application manifest, Windows should not redirect protected-location config reads or writes into `AppData\Local\VirtualStore`; non-elevated writes to `Program Files` should fail normally and then follow the explicit elevation path.

## Build and deployment

- Build flow: `build.cmd` boots the VS environment via `devenv.cmd`, redirects `TMP` and `TEMP` into a fresh per-build subdirectory under the user's temp area (falling back to `build\tmp\` only when no user temp path is available), configures `build\cmake\` through `cmake -S -B -G "Visual Studio 17 2022" -A x64`, then builds that tree through `cmake --build`.
  - Keeps generated solutions, dependency state, and object files under `build\cmake\` while placing the final executable and linked PDB under `build\`.
  - Lets MSBuild perform incremental per-source recompilation.
  - Compiles the Win32 app target from the telemetry sources, vendored ADLX sources, and `resources/SystemTelemetry.rc`.
  - Builds `src/board_gigabyte_siv.cpp` separately as a Common Language Runtime-enabled object library so it can bridge into the Gigabyte .NET assemblies.
  - Resolves `mscoree.lib` from the Visual Studio developer environment through `NETFXSDKDir` and disables the linker manifest because the `.rc` file already embeds the maintained application manifest.
