System Telemetry architecture

- `docs/`: project documentation. `docs/specifications.txt` tracks user-visible dashboard behavior, `docs/diagnostics.txt` tracks diagnostics CLI behavior and output requirements, `docs/project.txt` tracks project environment and engineering constraints, and `docs/architecture.txt` tracks structural and code-organization details.
- `src/main.cpp`: Win32 app entry point, single-instance startup, command-line diagnostics handling for `/trace`, `/dump`, `/screenshot`, `/reload`, `/fake`, `/scale`, and `/exit`, locale-stable fractional `/scale` parsing, monitor identification, per-monitor DPI handling, tray/menu actions, a shared 500 ms timer-driven refresh loop, executable-relative `config.ini` path resolution, elevation-aware `Save Config` saves, and orchestration of the telemetry runtime plus the dedicated dashboard renderer module.
- `src/config.h` + `src/config.cpp`: runtime config model plus load/save logic for executable-relative `config.ini`; it loads embedded resource defaults first, overlays executable-side config values on top, parses the compact `[display]` placement pair syntax plus the dedicated `[metric_scales]` section for metric-list normalization ceilings plus the `[layout]` and `[card.*]` layout language including the generic dashboard `cards = rows(...)` placement tree with `name:weight(...)` weight syntax plus the dedicated metric-row, drive-row, throughput, gauge, and widget-chrome geometry settings. Metric-list and drive-usage rows now derive their final packed heights from measured fonts plus config-driven bar-height and vertical-gap keys instead of from fixed row-height config values. The loader derives the rendered drive list from `drive_usage_list(...)` widget bindings, derives the requested named board temperature and fan metrics from layout references such as `board.temp.CPU`, `board.fan.CPU`, and `board.fan.System 1`, and saves only display placement and network adapter state while preserving the surrounding INI comments and structure. The config structs themselves now carry neutral zero/empty defaults so shipped fonts, colors, layout sizes, and metric-list scale ceilings come only from the embedded `resources/config.ini` template plus any executable-side `config.ini` overlay. The layout palette now includes a dedicated `graph_marker_color` used for throughput guide and time-marker lines.
- `src/dashboard_metrics.h` + `src/dashboard_metrics.cpp`: dashboard metric-binding layer that adapts `SystemSnapshot` into widget-oriented data shapes such as text values, gauge percentages, metric-list rows, throughput series, network footer text, clock text, and storage drive rows resolved by metric name. Metric-list rows now carry both the live fill ratio and a recent-peak ratio derived from retained normalized metric histories so the renderer can place translucent peak ghosts without duplicating telemetry-specific rules, and every non-capacity metric-list normalization ceiling comes from `AppConfig.metricScales` rather than from hard-coded constants. Throughput widgets keep the live numeric MB/s value from the latest sample but smooth the plotted history with a two-reading simple moving average before graph scaling and rendering; storage plots also switch their horizontal guide spacing from 5 MB/s to 50 MB/s when the averaged graph max rises above 50 MB/s so large ranges stay readable. A shared time-marker phase derived from the snapshot clock is attached to every throughput metric so 10-second vertical markers line up and scroll together across all throughput plots.
- `src/dashboard_renderer.h` + `src/dashboard_renderer.cpp`: dashboard layout and widget renderer module; it owns panel icon loading, UI font creation, font measurement, static layout resolution, widget binding parsing, and GDI/GDI+ drawing for cards and supported widget kinds. Metric-list and drive-usage row heights are computed during renderer initialization from measured font metrics plus config-driven bar-height and vertical-gap settings, and fixed-width text columns such as throughput labels, throughput axis labels, drive-letter labels, and drive-percent labels are likewise measured from the configured fonts plus config-driven padding instead of coming from separate hard-coded pixel literals. Renderer config geometry stays in logical units, while a render-scale factor derived from monitor DPI or headless `/scale` output multiplies fonts, spacing, window size, and screenshot dimensions into native device pixels before layout resolution. Metric and drive bars now render as antialiased pill capsules, while metric rows also overlay a translucent recent-peak capsule marker using the resolved metric history data. Widget bindings now carry a generic primary parameter string whose interpretation is owned by the hard-coded widget type, such as metric names for `text(...)`, `gauge(...)`, and `throughput(...)`, comma-separated metric refs with optional `=Label` overrides for `metric_list(...)`, or drive letters for `drive_usage_list(...)`; the renderer no longer carries compatibility aliases for retired widget names. Gauge, throughput, footer, and clock chrome spacing now come from layout config rather than from buried renderer constants. When a diagnostics trace stream is attached, the renderer also emits detailed layout-computation traces covering measured fonts, derived heights, resolved rectangles, and the active render scale.
- `src/utf8.h` + `src/utf8.cpp`: shared UTF-8 and UTF-16 conversion helpers used only where the UTF-8-first runtime model crosses Win32 or vendor API boundaries.
- `src/trace.h` + `src/trace.cpp`: shared trace wrapper around an optional output stream plus common trace-formatting helpers for Win32, PDH, ADLX, boolean, and numeric diagnostic fields.
- `src/telemetry.h` + `src/telemetry.cpp`: shared telemetry snapshot types plus the `TelemetryCollector` implementation that gathers Windows-native CPU brand-string, PDH, IP Helper, disk-capacity, system-wide disk-throughput, and clock data behind a small public API. After initialization, each `UpdateSnapshot()` call refreshes every live metric source on the shared 500 ms UI timer instead of keeping separate per-metric skip intervals, so CPU, GPU, memory, network, storage throughput, drive capacity, and clock stay aligned to one snapshot cadence. The snapshot now also retains normalized 60-sample metric histories for metric-list rows such as RAM, VRAM, temperature, clock, fan, and requested board sensors so the renderer can place recent-peak ghosts from the same 30-second cadence already used by throughput histories; those non-capacity history normalizations use the same config-driven metric-scale ceilings as the live metric rows. CPU board temperatures and fans now live only in the named `board.temperatures` and `board.fans` collections instead of duplicated `cpu.temperature` or `cpu.fan` snapshot fields.
- `src/telemetry_runtime.h` + `src/telemetry_runtime.cpp`: shared runtime abstraction plus the real telemetry runtime wrapper that adapts `TelemetryCollector` to the UI/diagnostics host flow. The same runtime layer also centralizes whether fake-runtime initialization failures are allowed to show modal dialogs so `/trace` diagnostics paths stay non-blocking.
- `src/telemetry_runtime_fake.cpp`: fake telemetry runtime implementation that reloads `telemetry_fake.txt` through the dump parser once per second for repeatable UI testing.
- `src/snapshot_dump.h` + `src/snapshot_dump.cpp`: versioned text dump serializer/parser for complete telemetry snapshots, including provider diagnostics, named board metrics, throughput plus metric-row histories, drive rows, and full dumped local time; the same format powers both `/dump` exports and `/fake` imports.
- `src/gpu_vendor.h` + `src/gpu_vendor.cpp`: vendor GPU provider abstraction plus factory selection.
- `src/gpu_amd_adl.cpp`: AMD GPU provider implementation using AMD ADLX for GPU name, temperature, clock, and fan speed.
- `src/board_vendor.h` + `src/board_vendor.cpp`: motherboard-vendor provider abstraction plus factory selection. The shared sample now carries only the board-provider metadata and named metrics that the dashboard and dump path actively use.
- `src/board_gigabyte_siv.cpp`: Gigabyte motherboard provider implementation that detects Gigabyte baseboards from registry-backed firmware metadata, launches `GigabyteSivProbe.exe /daemon` beside the running executable, keeps that helper process alive, parses framed key/value snapshots from its stdout stream, and maps the layout-requested named board fan and temperature metrics directly to SIV sensor titles without any CPU auto-selection layer.
- `src/GigabyteSivProbe.cs`: small helper executable built beside the app; it discovers Gigabyte SIV from the uninstall registry, loads the installed SIV assemblies, connects to the `GigabyteEngine` IPC pipe through `Gigabyte.Engine.EnvironmentControl.HardwareMonitor.HardwareMonitorControlModule`, queries fan and temperature data from the `HwRegister` source, and either prints one snapshot or, in `/daemon` mode, continuously streams framed `key=value` snapshots for the native provider.
- `resources/`: embedded UI assets, including the panel header icons extracted from the design sketch, the custom app icon, the self-documented embedded `config.ini` template, and the `.rc` plus resource-id definitions used to compile them into the executable.
- `resources/SystemTelemetry.manifest`: embedded Win32 application manifest that keeps the app at `asInvoker`, disables legacy file virtualization so config I/O targets the real executable-side `config.ini`, and opts the process into per-monitor DPI-aware rendering.
- `install.cmd`: elevated deployment script that builds via `build.cmd`, installs `build\SystemTelemetry.exe` plus `build\GigabyteSivProbe.exe` into `C:\Program Files\SystemTelemetry`, and registers a machine-wide `HKLM\Software\Microsoft\Windows\CurrentVersion\Run` entry for automatic startup on every user sign-in.
- `src/vendor/adlx/...`: vendored AMD ADLX headers/helper/runtime glue used by the AMD GPU provider.
- `devenv.cmd` + `devenv.txt`: local build-environment bootstrap plus its machine-specific setup contract; `devenv.cmd` prepares the Visual Studio toolchain and exports the `CSC` path used by the helper build, while `devenv.txt` documents that requirement for future machines.
- Startup flow: `wWinMain` opts into per-monitor-V2 DPI awareness, then parses `/trace`, `/dump`, `/screenshot`, `/reload`, `/fake`, `/scale`, and `/exit`. `/exit` routes into a headless diagnostics path that creates either the real telemetry runtime or the fake-file runtime, performs one update, optionally forces a config reload through the same runtime-reload helper used by the live UI, optionally writes the requested dump and screenshot files at the requested headless render scale, and exits without entering the single-instance GUI path. Without `/exit`, the normal dashboard UI path still starts, and any requested diagnostics outputs stay attached to that running instance.
- Telemetry flow: `DashboardApp` asks `TelemetryCollector` for a `SystemSnapshot`, then the window paints directly from that snapshot.
- Provider flow: `TelemetryCollector` owns both a vendor GPU provider and a motherboard-vendor provider. Windows collects generic metrics such as CPU load, CPU clock, GPU load, used VRAM, network, layout-configured drive capacity, system-wide storage read/write throughput, and time; total VRAM capacity comes from Windows adapter info as a fallback and may be overridden by a GPU vendor provider when the vendor API has a more reliable hardware-specific value, while named `board.temp.*` and `board.fan.*` metrics may be filled by a board-specific provider on the target Gigabyte system.
- No-WMI flow: the runtime telemetry path does not use WMI. The CPU name comes from native CPU brand-string discovery with a registry fallback, while GPU names fall back to built-in labels unless a vendor provider supplies a better name.
- Current vendor coverage: AMD Radeon GPUs use ADLX through `src/gpu_amd_adl.cpp`, and this target Gigabyte motherboard uses the SIV IPC-based board-fan path in `src/board_gigabyte_siv.cpp` plus `src/GigabyteSivProbe.cs`. Additional vendors can be added as new provider source files without complicating the Windows-native collector.
- Diagnostics output flow: `src/main.cpp` owns a small diagnostics session helper that opens `telemetry_trace.txt` in append mode when `/trace` is enabled, passes that stream through the active telemetry runtime, and separately overwrites `telemetry_dump.txt` and `telemetry_screenshot.png` on each requested output refresh. Dump files are emitted through `src/snapshot_dump.cpp` as machine-parseable snapshot text, and screenshot exports render through the effective runtime layout config so headless output matches the live dashboard styling. In UI mode those dump/screenshot refreshes run once per second alongside the normal repaint timer; in `/exit` mode they run once after the first telemetry update.
- Fake telemetry flow: `src/telemetry_runtime_fake.cpp` skips all live telemetry providers, reads `telemetry_fake.txt` through `src/snapshot_dump.cpp`, exposes that parsed snapshot directly to the renderer, and reloads the file once per second so manual edits can drive repeatable UI testing. Required fake-file open/parse failures now respect the shared diagnostics dialog policy so `/fake /exit` fails promptly under `/trace` instead of blocking on a modal dialog.
- Diagnostics tracing flow: `src/telemetry.cpp` traces CPU, GPU, memory, network, storage capacity, storage throughput, and PDH setup/refresh calls with returned Win32 or PDH statuses plus selected values; vendor providers such as `src/gpu_amd_adl.cpp` and `src/board_gigabyte_siv.cpp` add matching per-call result tracing for their API interactions.
- Network selection flow: `src/telemetry.cpp` combines `GetIfTable2` traffic counters with one `GetAdaptersAddresses` snapshot per refresh so it can trace and rank adapters by routed-connection signals such as default-gateway presence, IPv4 address availability, hardware/connector flags, and observed traffic before choosing the adapter to display; when `network.adapter_name` is configured, it prefers exact case-insensitive alias/description matches before falling back to substring matching.
- Trace ownership flow: `src/main.cpp` creates the diagnostics-session top-level trace wrapper for startup and output markers when `/trace` is enabled, `src/telemetry.cpp` owns a shared `Trace` instance for collector diagnostics, and vendor providers receive that same trace wrapper so all trace lines share one output policy and formatting helper set.
- Text model flow: config values, telemetry names, diagnostics, monitor identifiers, and UI text are stored as UTF-8 `std::string` in the app's own data structures. Wide strings are created only at Win32 or ADLX call sites through `src/utf8.cpp`.
- Config I/O flow: `src/config.cpp` reads and writes UTF-8 `config.ini` text through standard `std::ifstream` and `std::ofstream` primitives, strips an existing UTF-8 BOM on load for compatibility, loads the embedded `resources/config.ini` blob through the Win32 resource API as the baseline config, overlays any executable-side `config.ini` values on top, writes UTF-8 directly on save, parses the `[layout]` geometry, row, palette, and font keys, and parses the `[card.*]` layout expressions and widget bindings; each parsed layout expression replaces the previous tree so executable-side config cleanly overrides embedded defaults instead of appending duplicate cards or widgets, and the embedded resource remains the only shipped source of default layout/card/widget definitions.
- UI flow: `DashboardApp` owns the native window, tray icon, popup menu, move mode overlay, config reload/save actions, custom app-icon loading for the window and tray, tray double-click bring-to-front handling, and the active telemetry runtime. It tracks the current monitor DPI, converts saved logical placement into physical pixels for window moves, listens for `WM_DPICHANGED` and display-topology changes, and rebuilds renderer resources whenever the dashboard crosses onto a monitor with a different scale. The move overlay now measures the active renderer fonts at paint time, lays itself out from those scaled metrics instead of fixed pixel rectangles, and includes the current monitor scale alongside monitor name and logical position. `DashboardRenderer` owns panel-icon loading, UI font creation, the static resolved dashboard layout cache, and GDI/GDI+ drawing for cards plus generic widget kinds such as text, gauge, metric-list, throughput, drive-usage, clock, footer, and spacer widgets. `DashboardMetricSource` resolves metric names like `board.temp.CPU`, `board.fan.System 1`, `gpu.load`, or `storage.read` into the data shape each widget needs. The layout resolver supports both stretch-to-fill vertical stacks and top-packed `stack_top(...)` containers that use per-widget preferred heights, and the widget renderers source their metric-bar thickness, throughput plot chrome sizes, gauge preferred size, and drive row column widths from layout config instead of from buried pixel literals. `Reload Config` tears down the current telemetry runtime before creating a replacement so vendor APIs such as AMD ADLX can be reinitialized cleanly, then reapplies the current executable-side `config.ini`, recomputes the static card/widget rectangles, and repaints immediately.
- Embedded config flow: `resources/config.ini` is the checked-in self-documented template and is compiled into the executable; the running app reads that embedded copy first, overlays any executable-side `config.ini`, and when it creates a live config for the first time it starts from the embedded text verbatim so all comments are preserved.
- Elevated config-save flow: `src/main.cpp` probes whether the current process can write the executable-side `config.ini`; if not, `Save Config` writes the effective settings to a temporary INI under `%TEMP%`, relaunches the same executable with `/save-config <temp-path>` using the `runas` verb, waits for that elevated helper instance to rewrite the live `config.ini`, and then removes the temporary handoff file.
- Virtualization behavior: because the executable now carries an explicit application manifest, Windows should not redirect protected-location config reads or writes into `AppData\Local\VirtualStore`; non-elevated writes to `Program Files` should fail normally and then follow the explicit elevation path instead.
- Build flow: `build.cmd` boots the VS environment via `devenv.cmd`, redirects `TMP` and `TEMP` into a fresh per-build subdirectory under the user's temp area (falling back to `build\tmp\` only when no user temp path is available), runs `nmake`, and cleans any stray top-level `RC*`, `RD*`, or `CSC*.TMP` scratch files left by toolchain quirks. This keeps real build outputs in `build\` while letting MSVC tools delete short-lived scratch files even when the Codex sandbox disallows deleting files inside the workspace. `devenv.cmd` is responsible for exporting the machine-specific `CSC` compiler path, and `Makefile` fails early if that environment contract is missing. `Makefile` first compiles `resources/SystemTelemetry.rc` into `build/SystemTelemetry.res`, then builds the Win32 app, telemetry sources, vendor provider sources, vendored ADLX helper sources, and the C# `GigabyteSivProbe.exe` helper into `build\`, linking the resource blob so embedded icons ship inside the executable while compiler/linker byproducts such as `.obj`, `.pdb`, `.ilk`, and `.res` stay under `build\`.
