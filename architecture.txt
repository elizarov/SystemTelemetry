System Telemetry architecture

- `src/main.cpp`: Win32 app entry point, single-instance startup, `/dump` handling, monitor identification, tray/menu actions, timer-driven refresh loop, and all GDI rendering/layout.
- `src/config.h` + `src/config.cpp`: runtime config model plus load/save logic for `config.ini` in the launch directory; saves display placement, network adapter name, and Gigabyte channel-name selections while preserving other settings.
- `src/utf8.h` + `src/utf8.cpp`: shared UTF-8 and UTF-16 conversion helpers used only where the UTF-8-first runtime model crosses Win32 or vendor API boundaries.
- `src/trace.h` + `src/trace.cpp`: shared trace wrapper around an optional output stream plus common trace-formatting helpers for Win32, PDH, ADLX, boolean, and numeric diagnostic fields.
- `src/telemetry.h` + `src/telemetry.cpp`: shared telemetry snapshot types plus the `TelemetryCollector` implementation that gathers Windows-native CPU brand-string, PDH, IP Helper, disk, and clock data behind a small public API.
- `src/gpu_vendor.h` + `src/gpu_vendor.cpp`: vendor GPU provider abstraction plus factory selection.
- `src/gpu_amd_adl.cpp`: AMD GPU provider implementation using AMD ADLX for GPU name, temperature, clock, and fan speed.
- `src/board_vendor.h` + `src/board_vendor.cpp`: motherboard-vendor provider abstraction plus factory selection.
- `src/board_gigabyte_siv.cpp`: Gigabyte motherboard provider implementation that detects Gigabyte baseboards from registry-backed firmware metadata, launches `GigabyteSivProbe.exe /daemon` beside the running executable, keeps that helper process alive, parses framed key/value snapshots from its stdout stream, resolves configured Gigabyte fan and temperature channel names when provided, otherwise prefers `CPU`-named channels and only then falls back to the first usable reading, and exposes those resolved names back to telemetry for dumps and config persistence.
- `src/GigabyteSivProbe.cs`: small helper executable built beside the app; it discovers Gigabyte SIV from the uninstall registry, loads the installed SIV assemblies, connects to the `GigabyteEngine` IPC pipe through `Gigabyte.Engine.EnvironmentControl.HardwareMonitor.HardwareMonitorControlModule`, queries fan and temperature data from the `HwRegister` source, and either prints one snapshot or, in `/daemon` mode, continuously streams framed `key=value` snapshots for the native provider.
- `resources/`: embedded UI assets, including the panel header icons extracted from the design sketch, the custom app icon, and the `.rc` plus resource-id definitions used to compile them into the executable.
- `install.cmd`: elevated deployment script that builds via `build.cmd`, installs `build\SystemTelemetry.exe` plus `build\GigabyteSivProbe.exe` into `C:\Program Files\SystemTelemetry`, creates the live `config.ini` from `src\config.ini` on first install, grants Users modify access to that installed config file so runtime config saves still work under Program Files, and registers a machine-wide `HKLM\Software\Microsoft\Windows\CurrentVersion\Run` entry for automatic startup on every user sign-in.
- `src/vendor/adlx/...`: vendored AMD ADLX headers/helper/runtime glue used by the AMD GPU provider.
- `devenv.cmd` + `devenv.txt`: local build-environment bootstrap plus its machine-specific setup contract; `devenv.cmd` prepares the Visual Studio toolchain and exports the `CSC` path used by the helper build, while `devenv.txt` documents that requirement for future machines.
- Startup flow: `wWinMain` checks for `/dump` first; dump mode initializes `TelemetryCollector`, opens `telemetry_dump.txt` beside the executable as a single `std::ofstream`, passes that stream into telemetry collection so lower layers can trace to the same file, asks telemetry to write the final dump snapshot directly into that stream, then uses the dashboard renderer offscreen to paint that same snapshot into a bitmap and saves `telemetry_screenshot.png` beside the executable before exiting without entering the single-instance GUI path.
- Telemetry flow: `DashboardApp` asks `TelemetryCollector` for a `SystemSnapshot`, then the window paints directly from that snapshot.
- Provider flow: `TelemetryCollector` owns both a vendor GPU provider and a motherboard-vendor provider. Windows collects generic metrics such as CPU load, CPU clock, GPU load, used VRAM, network, drives, and time; total VRAM capacity comes from Windows adapter info as a fallback and may be overridden by a GPU vendor provider when the vendor API has a more reliable hardware-specific value, while CPU temperature and fan RPM may be filled by a board-specific provider on the target Gigabyte system.
- No-WMI flow: the runtime telemetry path does not use WMI. The CPU name comes from native CPU brand-string discovery with a registry fallback, while GPU names fall back to built-in labels unless a vendor provider supplies a better name.
- Current vendor coverage: AMD Radeon GPUs use ADLX through `src/gpu_amd_adl.cpp`, and this target Gigabyte motherboard uses the SIV IPC-based board-fan path in `src/board_gigabyte_siv.cpp` plus `src/GigabyteSivProbe.cs`. Additional vendors can be added as new provider source files without complicating the Windows-native collector.
- Dump tracing flow: `src/main.cpp` owns the dump file stream and writes high-level `/dump` progress markers as plain UTF-8 without a BOM, then passes that same stream through `TelemetryCollector` into vendor providers such as `src/gpu_amd_adl.cpp` and `src/board_gigabyte_siv.cpp` so all trace output and the final snapshot land in one file without custom file-append helpers; `TelemetryCollector::DumpText` also writes a structured board-provider debug block for the Gigabyte path, including the selected CPU temperature sensor and sampled temperature when available.
- Diagnostics tracing flow: `src/telemetry.cpp` traces CPU, GPU, memory, network, storage, and PDH setup/refresh calls with returned Win32 or PDH statuses plus selected values; vendor providers such as `src/gpu_amd_adl.cpp` and `src/board_gigabyte_siv.cpp` add matching per-call result tracing for their API interactions.
- Network selection flow: `src/telemetry.cpp` combines `GetIfTable2` traffic counters with one `GetAdaptersAddresses` snapshot per refresh so it can trace and rank adapters by routed-connection signals such as default-gateway presence, IPv4 address availability, hardware/connector flags, and observed traffic before choosing the adapter to display; when `network.adapter_name` is configured, it prefers exact case-insensitive alias/description matches before falling back to substring matching.
- Trace ownership flow: `src/main.cpp` creates the dump-mode top-level trace wrapper for startup markers, `src/telemetry.cpp` owns a shared `Trace` instance for collector diagnostics, and vendor providers receive that same trace wrapper so all trace lines share one output policy and formatting helper set.
- Text model flow: config values, telemetry names, diagnostics, monitor identifiers, and UI text are stored as UTF-8 `std::string` in the app's own data structures. Wide strings are created only at Win32 or ADLX call sites through `src/utf8.cpp`.
- Config I/O flow: `src/config.cpp` reads and writes UTF-8 `config.ini` text through standard `std::ifstream` and `std::ofstream` primitives, strips an existing UTF-8 BOM on load for compatibility, writes UTF-8 directly on save, and parses the optional `vendor.gigabyte.fan_channel` and `vendor.gigabyte.temperature_channel` selectors as channel names.
- UI flow: `DashboardApp` owns the native window, tray icon, popup menu, move mode overlay, config update action, panel rendering, embedded panel-header icon loading, custom app-icon loading for the window and tray, tray double-click bring-to-front handling, centered time/date text layout in the time panel, vertically stacked network up/down readouts with per-plot scale axes, 5 MB/s horizontal guide lines, and a shared footer line, and the shared usage-fill color that is reused by CPU/GPU gauges plus storage and metric bars so occupancy visuals stay consistent; the CPU/GPU gauge renderer paints a full track ring first and then overlays only the load-sized sliver in the usage color, the CPU panel includes thin-bar temperature and fan rows fed by the board provider when available, and `Update Config` persists the currently effective auto-selected adapter and Gigabyte channel names in addition to display placement.
- Source-tree config: `src/config.ini` is only the checked-in example template; the running app reads/writes `config.ini` beside the launched executable/working directory, and when it has to create that live file from scratch it writes only the active sections/keys rather than copying the example-header comments.
- Build flow: `build.cmd` boots the VS environment via `devenv.cmd`, redirects `TMP` and `TEMP` into a fresh per-build subdirectory under `build\tmp\`, runs `nmake`, and cleans any stray top-level `RC*`, `RD*`, or `CSC*.TMP` scratch files left by toolchain quirks. `devenv.cmd` is responsible for exporting the machine-specific `CSC` compiler path, and `Makefile` fails early if that environment contract is missing. `Makefile` first compiles `resources/SystemTelemetry.rc` into `build/SystemTelemetry.res`, then builds the Win32 app, telemetry sources, vendor provider sources, vendored ADLX helper sources, and the C# `GigabyteSivProbe.exe` helper into `build\`, linking the resource blob so embedded icons ship inside the executable while compiler/linker byproducts such as `.obj`, `.pdb`, `.ilk`, and `.res` stay under `build\`.
