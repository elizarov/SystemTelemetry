System Telemetry architecture

- `src/main.cpp`: Win32 app entry point, single-instance startup, `/dump` handling, monitor identification, tray/menu actions, timer-driven refresh loop, and all GDI rendering/layout.
- `src/config.h` + `src/config.cpp`: runtime config model plus load/save logic for `config.ini` in the launch directory; saves display name and relative position while preserving other settings.
- `src/utf8.h` + `src/utf8.cpp`: shared UTF-8 and UTF-16 conversion helpers used only where the UTF-8-first runtime model crosses Win32 or vendor API boundaries.
- `src/trace.h` + `src/trace.cpp`: shared trace wrapper around an optional output stream plus common trace-formatting helpers for Win32, PDH, ADLX, boolean, and numeric diagnostic fields.
- `src/telemetry.h` + `src/telemetry.cpp`: shared telemetry snapshot types plus the `TelemetryCollector` implementation that gathers Windows-native PDH, IP Helper, disk, and clock data behind a small public API.
- `src/gpu_vendor.h` + `src/gpu_vendor.cpp`: vendor GPU provider abstraction plus factory selection.
- `src/gpu_amd_adl.cpp`: AMD GPU provider implementation using AMD ADLX for GPU name, temperature, clock, and fan speed.
- `resources/`: embedded UI assets, including the panel header icons extracted from the design sketch, the custom app icon, and the `.rc` plus resource-id definitions used to compile them into the executable.
- `src/vendor/adlx/...`: vendored AMD ADLX headers/helper/runtime glue used by the AMD GPU provider.
- Startup flow: `wWinMain` checks for `/dump` first; dump mode initializes `TelemetryCollector`, opens `telemetry_dump.txt` beside the executable as a single `std::ofstream`, passes that stream into telemetry collection so lower layers can trace to the same file, asks telemetry to write the final dump snapshot directly into that stream, then uses the dashboard renderer offscreen to paint that same snapshot into a bitmap and saves `telemetry_screenshot.png` beside the executable before exiting without entering the single-instance GUI path.
- Telemetry flow: `DashboardApp` asks `TelemetryCollector` for a `SystemSnapshot`, then the window paints directly from that snapshot.
- Provider flow: `TelemetryCollector` owns a vendor GPU provider. Windows collects generic metrics such as CPU load, CPU clock, GPU load, used VRAM, network, drives, and time; total VRAM capacity comes from Windows adapter info as a fallback and may be overridden by a vendor provider when the vendor API has a more reliable hardware-specific value.
- No-WMI flow: the runtime telemetry path does not use WMI. CPU/GPU names fall back to built-in labels unless a vendor provider supplies a better name.
- Current vendor coverage: AMD Radeon GPUs use ADLX through `src/gpu_amd_adl.cpp`. Additional vendors can be added as new provider source files without complicating the Windows-native collector.
- Dump tracing flow: `src/main.cpp` owns the dump file stream and writes high-level `/dump` progress markers as plain UTF-8 without a BOM, then passes that same stream through `TelemetryCollector` into vendor providers such as `src/gpu_amd_adl.cpp` so all trace output and the final snapshot land in one file without custom file-append helpers.
- Diagnostics tracing flow: `src/telemetry.cpp` traces CPU, GPU, memory, network, storage, and PDH setup/refresh calls with returned Win32 or PDH statuses plus selected values; vendor providers such as `src/gpu_amd_adl.cpp` add matching per-call result tracing for their API interactions.
- Network selection flow: `src/telemetry.cpp` combines `GetIfTable2` traffic counters with one `GetAdaptersAddresses` snapshot per refresh so it can trace and rank adapters by routed-connection signals such as default-gateway presence, IPv4 address availability, hardware/connector flags, and observed traffic before choosing the adapter to display.
- Trace ownership flow: `src/main.cpp` creates the dump-mode top-level trace wrapper for startup markers, `src/telemetry.cpp` owns a shared `Trace` instance for collector diagnostics, and vendor providers receive that same trace wrapper so all trace lines share one output policy and formatting helper set.
- Text model flow: config values, telemetry names, diagnostics, monitor identifiers, and UI text are stored as UTF-8 `std::string` in the app's own data structures. Wide strings are created only at Win32 or ADLX call sites through `src/utf8.cpp`.
- Config I/O flow: `src/config.cpp` reads and writes UTF-8 `config.ini` text through standard `std::ifstream` and `std::ofstream` primitives, strips an existing UTF-8 BOM on load for compatibility, and writes UTF-8 directly on save.
- UI flow: `DashboardApp` owns the native window, tray icon, popup menu, move mode overlay, config update action, panel rendering, embedded panel-header icon loading, custom app-icon loading for the window and tray, tray double-click bring-to-front handling, centered time/date text layout in the time panel, vertically stacked network up/down readouts with per-plot scale axes and a shared footer line, and the shared usage-fill color that is reused by CPU/GPU gauges plus storage and metric bars so occupancy visuals stay consistent; the CPU/GPU gauge renderer paints a full track ring first and then overlays only the load-sized sliver in the usage color.
- Source-tree config: `src/config.ini` is only the checked-in example template; the running app reads/writes `config.ini` beside the launched executable/working directory.
- Build flow: `build.cmd` boots the VS environment via `devenv.cmd` and runs `nmake`; `Makefile` first compiles `resources/SystemTelemetry.rc` into `build/SystemTelemetry.res`, then builds the Win32 app, telemetry sources, vendor provider sources, and vendored ADLX helper sources into `build/SystemTelemetry.exe`, linking the resource blob so embedded icons ship inside the executable while compiler/linker byproducts such as `.obj`, `.pdb`, `.ilk`, and `.res` stay under `build\`.
